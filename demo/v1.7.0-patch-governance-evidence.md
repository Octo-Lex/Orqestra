# v1.7.0 Agent Review Reliability & Patch Application Governance Evidence

## Purpose

Make the AI proposal → human review → file application boundary airtight before expanding agent scope.

## Critical Gaps Found and Fixed

| Gap | Fix |
|-----|-----|
| `handleAccept` only sets React state, doesn't apply patch | New `apply_agent_patch_cmd` with full validation |
| `write_file_cmd` has zero validation | `PatchApplicationGuard` validates every path and content |
| No forbidden-path check at write time | Server-side + forbidden-pattern enforcement |
| No audit trail | Append-only JSONL at `.Orqestra/agents/{type}/audit.jsonl` |
| No before/after checksums | Content + checksum verification before and after write |
| `AgentRunner` has auto_commit path | Removed — agent runner no longer writes files |
| No proposal_id for correlation | Every patch has stable `proposal_id` |

## New Types

| Type | Purpose |
|------|---------|
| `PatchProposal` | Typed DTO with proposal_id, path, before/after, checksums |
| `PatchApplicationResult` | Durable outcome (proposed/rejected/apply_failed/applied) |
| `AgentType` | Server-side policy discriminator (docs/bugfix) |
| `AuditRecord` | JSONL audit record with all metadata |

## Server-Side Agent Policies

| Agent | Allowed Paths (server-side) | Frontend Role |
|-------|----------------------------|---------------|
| docs-agent | README.md, docs/, roadmap/, CHANGELOG.md | May narrow, not widen |
| bugfix-agent | All non-forbidden paths (scope from user selection) | May narrow, not widen |

## Forbidden Path Categories

1. Secret-risk: `.env`, `*.pem`, `*.key`, `id_rsa`, `id_ed25519`, `secrets.*`, `credentials.*`
2. Workflow-risk: `.github/workflows/`, `.github/actions/`
3. Binary: images, executables, archives, fonts, media
4. Dependency locks: `Cargo.lock`, `package-lock.json`, `yarn.lock`, etc.
5. Infrastructure: `docker-compose*`, `Dockerfile`

## State Machine

```
proposed → applied (after validation + atomic write + checksum)
proposed → rejected (user rejects, no file change)
proposed → apply_failed (validation fail, no file change)
```

"Accepted" is a UI state between proposed and applied — not recorded in audit until the Rust command succeeds.
